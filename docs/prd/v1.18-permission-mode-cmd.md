## Problem

Claude CLI's terminal UI shows the current permission mode (e.g. `bypass permissions on`, `plan mode on`) and supports `shift+tab` to cycle through them. claudeD has no equivalent тАФ users can't see what mode they're in or switch it mid-session. SDK supports runtime switching via `ClaudeSDKClient.set_permission_mode(...)`.

## User decisions (DDD 5/16, locked)

| # | Decision |
|---|---|
| 1 | Use candidate 3: `/mode` group with `set` / `cycle` / `current` subcommands (parallel to `/model`) |
| 2 | Footer: single new line (not appended to the cost row) |
| 3 | `/mode set` and `/mode cycle` reply with ephemeral confirmation embed |
| 4 | **Persistent**: user-explicit mode survives bot restart (saved to `data/sessions.json`). Unset тЖТ use `CLAUDED_PERMISSION_MODE` env / `"default"` fallback |
| 5 | Only expose 4 modes the CLI uses: `default`, `acceptEdits`, `plan`, `bypassPermissions`. `dontAsk` / `auto` not surfaced |
| 6 | Cycle order: `default тЖТ acceptEdits тЖТ plan тЖТ bypassPermissions тЖТ default` |

тЪая╕П **Decision #4 contrast with #210**: #210 made `model_override` *ephemeral* because `stored.model` was already polluted with forced-sonnet values from pre-#199 builds, so we couldn't trust the stored value. `permission_mode` has no such legacy pollution тАФ fresh schema field, no prior writes. User's underlying intent ("шо╛ч╜оф║Жх░▒чФишо╛ч╜оф║Ж") therefore applies directly without the cleanup safeguard.

## Goals

1. **`/mode` group** with 3 subcommands:
   - `/mode set <mode>` тАФ explicit set, autocomplete to 4-mode set
   - `/mode cycle` тАФ advance one step in fixed order
   - `/mode current` тАФ show current mode + source (override / env / default)
2. **Bridge runtime switch** тАФ call SDK's `set_permission_mode()`; track in `bridge._permission_mode_override` (new field).
3. **Footer line** тАФ when current effective mode != `"default"`, append a separate line below the cost row showing the mode (e.g., `ЁЯФТ plan` / `тЪб bypassPermissions`). When default, no extra line.
4. **Persistence** тАФ `data/sessions.json` row gains `permission_mode_override: str | None` field. Written via `save_session_state` when user has explicitly set it. Read on auto-resume / `/session resume` and re-injected as `SessionConfig.permission_mode_override` for the next bridge init.
5. **`/health` + `/session info`** тАФ include the current permission mode in their existing displays.

## Non-goals

- Surfacing `dontAsk` or `auto` SDK modes (per decision #5)
- Cross-channel "global" mode setting
- Hooking SDK's hook system for fine-grained per-tool decisions
- Re-implementing `CLAUDED_PERMISSION_MODE` env semantics тАФ keep as-is

## Design

### Tier resolution (mirror of `/model`'s 4-tier from #198 PRD)

| Tier | Source | Lifetime | Footer label |
|---|---|---|---|
| 1 | `/mode set <X>` тЖТ `_permission_mode_override` | persisted across restart | `{X}` (no suffix) |
| 2 | `CLAUDED_PERMISSION_MODE` env | persistent (env-set) | `{X} (env)` |
| 3 | unset тЖТ SDK / config default `"default"` | implicit | (no footer line) |

Cycle reads the **current effective mode** (override or env or default) and advances. Persists the new value as override.

### Footer

`discord_renderer.py` cost-footer builder. After the existing line:
```
-# ЁЯТ░ $0.0042 тФВ ЁЯУе 1.2k тФВ ЁЯУд 800 тФВ тП▒я╕П 3.4s тФВ ЁЯза 12%
```
add a conditional second line **only when effective mode != "default"**:
```
-# ЁЯФТ plan
```
or
```
-# тЪб bypassPermissions
```

Emoji mapping (centralized):
- `default` тЖТ (no footer line shown)
- `acceptEdits` тЖТ `тЬПя╕П acceptEdits`
- `plan` тЖТ `ЁЯФТ plan`
- `bypassPermissions` тЖТ `тЪб bypassPermissions`

### SDK runtime switch

`claude_bridge.py`:
```python
async def set_permission_mode(self, mode: str) -> None:
    """Runtime switch via SDK control-plane. Persists to _permission_mode_override
    so subsequent reads via .effective_permission_mode reflect the new value."""
    if self._client is None or not self._active:
        raise RuntimeError("bridge not active")
    await self._client.set_permission_mode(mode)
    self._permission_mode_override = mode
```

### Persistence schema

`data/sessions.json` per-thread row:
```json
{
  "session_id": "...",
  "project_path": "...",
  "system_prompt": "...",
  "model": null,
  "permission_mode_override": "plan",   // тЖР NEW; null when unset
  "last_active": "..."
}
```

`SessionStore.save_session(thread_id, session_id, project_path, *, model=None, system_prompt=None, permission_mode_override=None)` gains the new kwarg. `SessionManager.save_session_state` writes `bridge.permission_mode_override` (new accessor тАФ parallel to `bridge.explicit_model_override` from #199).

Read path: `bot.py:_handle_thread_message` resume block reads `stored.get("permission_mode_override")` and feeds it into `SessionConfig(permission_mode_override=...)`. New bridge instances honor it at `start()`.

### Cycle command

Pure helper:
```python
_CYCLE_ORDER = ["default", "acceptEdits", "plan", "bypassPermissions"]

def _next_mode(current: str) -> str:
    try:
        idx = _CYCLE_ORDER.index(current)
    except ValueError:
        return "default"  # unknown / dontAsk / auto тЖТ snap to start
    return _CYCLE_ORDER[(idx + 1) % len(_CYCLE_ORDER)]
```

## Acceptance criteria

- [ ] `/mode cycle` advances through 4 modes in order; loops back to default
- [ ] `/mode current` shows current mode + source tag (override / env / default)

## Out of scope

- Surfacing `dontAsk` / `auto` SDK modes
- Per-channel global mode policy
- Hook-based fine-grained tool permission decisions
- Migration of legacy fields in `sessions.json`

## Risks

- **SDK type drift**: `PermissionMode` is a `Literal` тАФ if SDK renames a mode, our hardcoded `_CYCLE_ORDER` + `MODE_EMOJI` keys silently miss. Mitigation: a contract pin test like #193's (`from claude_agent_sdk.types import PermissionMode` + assert keys are subset of Literal values).
- **Default-mode resume edge case**: user explicitly does `/mode set default`. Is that "they want default explicitly" or "unset"? Per decision #4 + #210 lesson: explicit-default is still an explicit override (persisted as `"default"`, not `None`). Footer still omits (default = no footer line). Distinguishable by `/mode current` showing source = override.
